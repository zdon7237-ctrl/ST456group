## 5. 已确认问题

### 问题 1：主训练损失覆盖整个输入序列，与条件生成任务定义不严格对齐，且与辅助目标的 mask 策略不一致

- 严重度：高
- 影响：
  - 当前实现会让主语言模型损失覆盖 `prompt + [TARGET] + target` 全段，而不只覆盖目标续写段落。
  - proposal 将任务写成“给定前 `k` 段，输出下一段”，因此当前实现与一种更严格的条件生成读法并不完全一致。
  - 更重要的是，当前主损失和辅助目标的监督边界并不一致：主损失保留 prompt token，辅助候选目标却显式 mask 掉 prompt。
  - 由于训练器又配置了 `load_best_model_at_end=True` 且 `metric_for_best_model="eval_loss"`，这个选择会直接影响验证损失和最佳 checkpoint 的选择。
- 证据：
  - proposal 将任务定义为“Input consists of the first `k` paragraphs; Output is the target next paragraph; Token-level cross-entropy is used as the basic loss function”：`proposal.md:31-34`
  - 主训练 collator 在 `_encode_feature()` 中构造 `full_input_ids` 后，直接令 `labels = list(full_input_ids)`，没有对 prompt 部分做 `-100` mask：`codex-novel-continuation/src/novel_continuation/trainer_runtime.py:92-105`
  - 同一文件里，辅助候选目标 `_encode_candidate()` 却对 prompt 部分做了显式 mask：`codex-novel-continuation/src/novel_continuation/trainer_runtime.py:107-110`
  - 训练器构建时开启了 `load_best_model_at_end=True` 并固定按 `eval_loss` 选择最佳模型：`codex-novel-continuation/src/novel_continuation/training.py:521-545`
- 是否影响 Colab 主线：
  - 会影响。
  - 这不是“代码会跑不起来”的阻断错误，但它会影响 E1-E5 的训练目标解释、验证损失含义和 checkpoint 选择，属于结果有效性层面的高优先级问题。

### 问题 2：当前 PPL 指标的构造方式不适合在 E1-E5 之间直接横向比较

- 严重度：中
- 影响：
  - 当前 PPL 不是在 `generated_text` 上计算，而是在 `prompt + gold_target` 上计算，因此它衡量的是模型对参考目标文本的条件似然，而不是生成样本本身的质量。
  - `compute_perplexity()` 在调用模型时直接使用 `labels=encoded["input_ids"]`，没有对 prompt 部分做 `-100` mask，因此 prompt token 也进入 loss。
  - E1 使用 `context_size=2, plain`，E3 使用 `context_size=4, structured`；而 `build_prompt()` 在 `structured` 模式下还会加入 `[PARAGRAPH i]` 标记，因此不同实验的 prompt 长度和结构本身存在系统差异。
  - 在这种构造下，PPL 数值会同时混入参考目标的可预测性和 prompt 的长度/格式差异，不适合作为 E1-E5 的直接 pairwise 横向比较指标。
- 证据：
  - `compute_perplexity()` 调用模型时直接传入 `labels=encoded["input_ids"]`：`codex-novel-continuation/src/novel_continuation/evaluation.py:89-95`
  - `evaluate_generated_rows()` 组装 PPL 输入时使用的是 `prompt + [TARGET] + gold_target`：`codex-novel-continuation/src/novel_continuation/evaluation.py:148-150`
  - E1 配置为 `context_size: 2` 且 `context_format: plain`：`codex-novel-continuation/configs/e1_distilgpt2_plain_full.yaml:1-10`
  - E3 配置为 `context_size: 4` 且 `context_format: structured`：`codex-novel-continuation/configs/e3_distilgpt2_structured_long_context.yaml:1-10`
  - `build_prompt()` 在 `structured` 模式下会插入 `[PARAGRAPH i]` 标记：`codex-novel-continuation/src/novel_continuation/prompting.py:15-30`
- 是否影响 Colab 主线：
  - 会影响。
  - 这不会阻断 Colab 运行，但会直接影响 E1-E5 的 PPL 表格如何解读；若不写 caveat，最终报告里的 PPL 对比很容易被质疑。

### 问题 3：Human evaluation 导出脚本声明为 blinded，但实际导出了 `gold_target`

- 严重度：中
- 影响：
  - 人工评估表中包含参考答案 `gold_target`，评审者在打分时可能被标准答案锚定，影响对 contextual coherence、character consistency、fluency 的独立判断。
  - 这会削弱“blind human evaluation”的可信度。
- 证据：
  - 脚本文档字符串和 CLI 描述都写的是“blinded CSV”：`codex-novel-continuation/scripts/prepare_human_eval.py:1`, `codex-novel-continuation/scripts/prepare_human_eval.py:78-80`
  - 但 `build_human_eval_rows()` 实际把 `gold_target` 放进导出行：`codex-novel-continuation/scripts/prepare_human_eval.py:39-59`
  - CSV 字段列表也显式包含 `gold_target`：`codex-novel-continuation/scripts/prepare_human_eval.py:62-75`
  - 现有测试同样把 `gold_target` 当作导出契约的一部分：`codex-novel-continuation/tests/evaluation/test_metrics.py:68-94`
  - proposal 中的人评目标是 contextual coherence 与 character consistency：`proposal.md:51-56`
- 是否影响 Colab 主线：
  - 不会阻断 Colab 运行。
  - 但如果当前 CSV 直接发给组员打分，会影响人工评估结果的说服力。

### 问题 4：`selection_metric` 配置项当前未真正生效，trainer 实际固定按 `eval_loss` 选最佳模型

- 严重度：中
- 影响：
  - 配置文件和默认配置中都声明了 `selection_metric: perplexity`，但训练器构建时没有读取该配置，而是固定使用 `metric_for_best_model="eval_loss"`。
  - 这意味着当前配置项与实际行为不一致，属于“配置契约漂移”。
  - 报告中如果写成“按配置中的 perplexity 选择最佳模型”，将与实现不一致。
- 证据：
  - 默认训练配置声明了 `selection_metric: perplexity`：`codex-novel-continuation/src/novel_continuation/training.py:18-31`
  - 主实验配置同样写了 `selection_metric: perplexity`：`codex-novel-continuation/configs/e1_distilgpt2_plain_full.yaml:1-22`
  - 但 Hugging Face `TrainingArguments` 被硬编码为 `metric_for_best_model="eval_loss"`：`codex-novel-continuation/src/novel_continuation/training.py:521-545`
- 是否影响 Colab 主线：
  - 不会阻断 Colab 运行。
  - 对当前实验结果的排序影响可能有限，因为在同一标签定义下 `eval_loss` 与 perplexity 单调对应；但配置项名和实际行为不一致，最终报告的表述需要谨慎。

## 6. 风险提示（非主线阻断，但需要在最终报告中写清楚）

### 风险 1：`compute_perplexity()` 的聚合方式不是 token-weighted 聚合

- 证据：
  - `compute_perplexity()` 逐条读取 `outputs.loss.item()` 后，直接做样本级平均，再取 `exp`：`codex-novel-continuation/src/novel_continuation/evaluation.py:82-95`
- 解释：
  - `outputs.loss` 已经是单条样本上的平均 loss；当前实现再做一次样本级算术平均，等于让 50 token 和 250 token 的样本拥有相同权重。
  - 这会影响最终 PPL 数值的解释，尤其是在样本长度差异较大时更明显。
- 对 Colab 主线的影响：
  - 不阻断运行。
  - 但会进一步削弱当前 PPL 数值作为横向比较指标的说服力。

### 风险 2：`entity_overlap` 的实体提取规则误报概率较高，更接近启发式诊断量

- 证据：
  - 实体正则为 `\\b[A-Z][a-z]+(?:\\s+[A-Z][a-z]+)*\\b`：`codex-novel-continuation/src/novel_continuation/evaluation.py:11`
  - `compute_entity_overlap()` 直接对该正则抽取结果做交集比值：`codex-novel-continuation/src/novel_continuation/evaluation.py:49-59`
- 解释：
  - 这个规则会把大量句首普通词也当作“实体”，例如以大写开头的连接词、代词或普通名词。
  - 因此 `entity_overlap` 更像一个轻量级启发式诊断指标，不应被表述为可靠的角色一致性或命名实体一致性指标。
- 对 Colab 主线的影响：
  - 不阻断运行。
  - 主要影响最终报告中对 `entity_overlap` 的解释强度。

### 风险 3：当前 ROUGE-L 是自实现 LCS 变体，不等同于标准库版本

- 证据：
  - `_tokenize()` 仅做小写化和 `\\w+` 分词：`codex-novel-continuation/src/novel_continuation/evaluation.py:14-15`
  - `compute_rouge_l()` 直接基于手写 LCS 和 F1 公式计算：`codex-novel-continuation/src/novel_continuation/evaluation.py:18-46`
- 解释：
  - 当前实现没有 stemming，也没有与常见 ROUGE 库完全对齐的预处理或统计细节。
  - 因此最终报告若引用该指标，最好明确说明这是仓库内自实现的 ROUGE-L 变体。
- 对 Colab 主线的影响：
  - 不阻断运行。
  - 主要影响与外部文献或标准工具输出进行数值对照时的可比性。

### 风险 4：数据集切分按书内顺序进行，test 集会偏向各书后段样本

- 证据：
  - `assign_split()` 采用按样本下标的 80/10/10 顺序切分：`codex-novel-continuation/src/novel_continuation/dataset_builder.py:35-53`
  - `build_dataset.py` 又是在 `for source in SHERLOCK_SOURCES` 中逐本处理，并对每本书内部样本分别调用 `assign_split()`：`codex-novel-continuation/scripts/build_dataset.py:20-44`
- 解释：
  - 这不是代码错误，但它意味着每本书末尾一段比例的样本会稳定落入 test 集。
  - 因此最终报告在解释 generalization 时，应把它表述为一种时序切分/顺序切分设计，而不是完全随机切分。
- 对 Colab 主线的影响：
  - 不阻断运行。
  - 主要影响数据划分方法的解释与结论外推边界。

### 风险 5：E4 的 LoRA 模块范围包含 FFN 相关模块，参数量解释需要额外说明

- 证据：
  - LoRA 配置写为 `target_modules=["c_attn", "c_proj", "c_fc"]`：`codex-novel-continuation/src/novel_continuation/training.py:124-131`
- 解释：
  - 当前 E4 并不是一个仅作用于 attention 线性层的最小 LoRA 配方。
  - 如果最终报告要写“LoRA 使用更少参数”或“LoRA 更轻量”，最好配套实际可训练参数量说明，否则 E3/E4 对比的解释会不完整。
- 对 Colab 主线的影响：
  - 不阻断运行。
  - 主要影响 LoRA 对比结论的严谨性。

### 风险 6：`build_dataset.py` 的默认 `--context-size` 为 3，但 Colab 主线文档要求用 4

- 证据：
  - 构建脚本默认值是 `3`：`codex-novel-continuation/scripts/build_dataset.py:53-60`
  - Colab guide 明确要求运行 `--context-size 4`：`codex-novel-continuation/docs/colab-run-guide.md:36-41`, `codex-novel-continuation/docs/colab-notebook-guide.md:13-18`
- 解释：
  - 当前 Colab 主线 notebook 和 guide 都显式传了 `4`，因此主线运行不会因此出错。
  - 但如果有人脱离 notebook、直接按脚本默认值构建数据，实验设置会静默漂移。
- 对 Colab 主线的影响：
  - 当前主线不受影响。
  - 对手动复现实验的同学有潜在风险。

## 8. 建议的后续检查顺序

以下顺序只用于后续复核优先级，不包含代码修改方案：

1. 先决定最终报告如何处理当前 PPL 指标：是暂不做 E1-E5 横向对比，还是保留表格但显式写清 `prompt + gold_target`、prompt 参与 loss、以及非 token-weighted 聚合这三层 caveat。
2. 再复核训练主损失的监督边界，并在最终报告中明确说明为什么当前实现采用这一策略，以及它与辅助目标 mask 策略之间的关系。
3. 再复核 human evaluation 的导出协议，确认是否允许评审者看到 `gold_target`。
4. 再核对最终报告中关于“best model 选择依据”的措辞，不要把当前实现写成“按配置中的 perplexity 自由切换”。
5. 对 `entity_overlap`、自实现 ROUGE-L、数据集切分策略和 LoRA 模块范围这四类方法论 caveat，提前准备简洁说明，避免在最终报告里被当成默认标准做法。
6. 最后补强本地或 Colab 中的测试可执行性，尤其是 `generate_samples` 相关测试夹具与当前实现的同步问题，并继续保持 Colab 文档中的 `build_dataset.py --context-size 4` 用法。

但同时，当前实现存在会影响实验解释和报告可信度的已确认问题，尤其是：

- 主训练损失与条件生成任务定义不严格对齐，且与辅助目标的 mask 策略不一致
- 当前 PPL 指标的构造方式不适合直接做 E1-E5 横向比较
- human-eval 导出并非严格盲评
- 某些配置项与真实训练行为存在漂移

因此，当前代码状态可以继续跑实验，但不建议在不说明这些 caveat 的前提下，直接把所有结果当作“已完全对齐任务定义、评测协议和配置契约”的最终结论。

若最终报告仍保留 PPL 对比表，应显式写出该指标的构造 caveat；否则，这部分结果很容易因为可比性问题被直接质疑。
